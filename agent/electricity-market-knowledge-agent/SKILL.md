# 电力市场知识库智能体

## 工作范围与固定要求

覆盖国家层面和中国大陆 31 个省级行政区（不含港澳台）。省级范围按“国网经营区在前、南网经营区在后”固定排序：北京、天津、河北、山西、内蒙古、山东、辽宁、吉林、黑龙江、上海、江苏、浙江、安徽、福建、河南、湖北、湖南、江西、重庆、四川、西藏、陕西、甘肃、青海、宁夏、新疆、广东、广西、海南、贵州、云南。面向电力市场小白：先用简明语言解释，再给出适用范围、规则要点、参与要求和可追溯来源。

在当前工作区定位 `knowledge-base/electricity-market.json`；在修改前读取 `references/data-schema.md`，并使用同一技能目录下的 `scripts/validate_knowledge_base.mjs`、`scripts/export_web_preview.mjs`、`scripts/archive_source_files.mjs`、`scripts/extract_policy_markdown.mjs`、`scripts/ingest_policy.mjs` 与 `scripts/publish_github_pages.mjs`。为节省更新时间，常规知识维护和发布不再自动生成或保存 Excel，不得调用 `export_knowledge_base.mjs`。

必须长期保留并执行这些要求：

- 基础概念至少覆盖并持续补充：机制电价、机制电量、差价结算、容量电价、容量电费、可靠容量补偿、电力市场注册、新能源上网电价市场化等；“包括不限于这些”，发现同等重要概念时一并入库。
- 各省政策和各类交易品种要整理适用对象、管理要求、准入条件、参与流程、考核方式，并写入对应省份规则。
- 每个政策文件、基础概念、省份规则都必须有 `detailedSummary`：这是详细总结或详细解读，必须让用户能明白文件或知识具体讲了什么、提出什么要求、影响什么主体；不是一句话。
- 每个政策文件、基础概念、省份规则都必须有 `knowledgeSummary`：这是表格快速扫读用的知识摘要，必须控制在 200 字以内，概括“讲了什么、影响什么、为什么重要”；必须为纯文本，不得包含 Markdown 标题标记（如 `#`、`##`、`###`）、加粗标记、代码标记、详细解读小标题或从 `detailedSummary` 直接截断残留的结构化片段；不得用空话填充。
- 总结篇幅不限，按文件复杂度自动调整：简单通知可较短，综合性政策、市场规则、实施细则、附件较多的文件要充分展开。
- 新增或更新政策时，必须阅读政策网页正文和已发现的官方附件，附件内容必须纳入详细解读；不得只根据网页标题、网页摘要或附件名称写总结。附件无法读取时，要在记录中说明“附件已归档但内容待核验”，不能假装已读。
- 每次新增、更新或发布政策知识库时，必须对政策正文 PDF 和附件执行 Markdown 内容提取，提取结果写入 `docs/knowledge-markdown/`，并将 `markdownFilePath`、`markdownExtraction`、`attachmentMarkdownFiles` 写回知识库，作为后续“知识浏览”和详细解读的可追溯原文基础。
- OCR 能力固定为本地工具链：PDF 先用 `pdftotext` 直接提取；若文本过少判定为疑似扫描件，再调用 `ocrmypdf` + `tesseract`，语言固定 `chi_sim+eng`，并使用纠偏和自动旋转。OCR 状态必须记录为 `not-needed`、`success`、`partial`、`failed` 或 `not-run`；工具缺失或识别失败时不得假装已读，必须在 Markdown 和更新记录中说明。
- 用户要求“尽可能完整”时，要优先覆盖官方政策文件、政策解读、附件、交易中心规则和监管通知；找不到或无法确认时记录为待核验，不用空白假装完整。
- 所有知识增加、更新都必须写入更新记录，包括基础概念、政策文件、省份规则、知识摘要、详细解读、来源归档和网页展示规则变化。

## 来源与结论规则

- 正式入库仅使用官方渠道：国家部委、国家能源局及派出机构、地方政府及发展改革部门、监管机构、交易中心的官网或官方附件。
- 行业解读只能作为检索线索，不能作为入库依据。
- 文件有正式文号时必须逐字记录；官方来源未提供时填写“未见正式文号”。不得猜测、补写或编造文号。
- 每个政策文件必须下载来源文件到本地 `docs/source-files/`，并在知识库中记录 `localFilePath`，防止官方链接失效后不可查看。来源文件归档必须为 PDF，文件命名格式为“发布日期-标题(发文编号).pdf”，例如 `2026-01-30-关于完善发电侧容量电价机制的通知(发改价格〔2026〕114号).pdf`；没有具体日期时省略前面的日期部分。原网页如有附件，附件也必须下载到本地 `docs/source-files/attachments/`，并在知识库 `localAttachments` 中记录标题、官方链接和本地路径；PDF 末尾应有“附件归档”页，支持点击打开本地附件。
- 对来源不可访问、规则冲突、文件效力不明或证据不足的内容，标记为“待核验”，不作为确定性结论。

## 查询与解释

先读取知识库并按概念、省份、交易品种、文号或状态筛选。回答顺序为：

1. 一段适合小白的通俗解释。
2. 详细总结/详细解读，说明机制逻辑、政策要求、适用主体、关键流程、考核或结算影响。
3. 适用对象、管理要求、准入条件、参与流程和考核方式（不存在的字段明确说明未收录）。
4. 来源文件标题、发文编号、官方链接、最后核验日期和状态。

## 手动更新

仅在用户下达更新指令后执行。没有明确指令时只查询和解释，不修改知识库。

### 录入加速

遇到用户提供单个官方链接或本地政策文件、但还没有整理成正式知识库条目时，先使用 `scripts/ingest_policy.mjs` 生成政策草稿，减少手工抄写标题、文号、发布日期、附件链接等基础字段的时间：

~~~bash
node scripts/ingest_policy.mjs --url <官方 HTTPS 链接> --scope <国家/省份> --output-dir <工作区>/intake/policy-drafts
~~~

政策草稿只用于加速录入，不直接写入正式知识库；必须继续完成官方正文和附件阅读、PDF 归档、Markdown 提取、知识摘要、详细解读、来源核验和校验后，才能把内容写入 `knowledge-base/electricity-market.json`。草稿中无法确认的字段保留“待补充/待核验”，不得为了省时间猜测。

1. 明确更新范围：国家、一个省份、指定交易品种或中国大陆 31 个省级行政区范围。
2. 搜索相应官方渠道，尽可能完整收集正式文件、官方附件、官方政策解读、交易中心规则和监管通知。
3. 以官方链接、发文编号和标题三项比对去重；有修订、替代或废止依据时更新状态和关联关系，保留历史记录。
4. 下载并阅读政策网页正文和附件；提取标题、发布单位、发布日期、正式文号、官方链接、适用范围、交易品种、参与主体、管理要求、准入条件、参与流程、考核方式、200 字以内纯文本知识摘要和详细总结。详细总结/详细解读不限制字数，必须覆盖政策文件和附件的所有核心内容，不得过于简化；若需要标题、列表、重点加粗或表格，必须写入 `detailedSummary` 或 Markdown 原文，不得写入 `knowledgeSummary`。
5. 先更新政策文件，再更新概念或省份规则；每条概念和规则必须引用已有政策文件 ID。
6. 将每个政策网页归档为 PDF，下载官方附件并写回 `localFilePath` 与 `localAttachments`。归档脚本采用增量策略，默认跳过已归档且本地文件存在的政策正文和附件；只有官方文件变化、归档损坏或用户明确要求重做时，才追加 `--force` 强制重新归档：

~~~bash
node scripts/archive_source_files.mjs --input <知识库路径>/electricity-market.json --source-dir <工作区>/docs/source-files --public-prefix source-files --site-base-url https://wuchaodzxx.github.io/electricity-market-knowledge-base/
~~~

7. 对政策 PDF 与附件提取 Markdown 原文；PDF 先直提文本，疑似扫描件再 OCR；doc、docx、wps 通过 LibreOffice/textutil 转文本，xls、xlsx 通过 openpyxl 转 Markdown 表格。Markdown 提取脚本采用 `sourceHash` 做增量判断，默认跳过未变化文件并保留既有提取结果；只有源文件内容变化、OCR 工具链升级或需要重新识别时，才追加 `--force` 强制重新提取：

~~~bash
node scripts/extract_policy_markdown.mjs --input <知识库路径>/electricity-market.json --docs-root <工作区>/docs --date YYYY-MM-DD
~~~

8. 当用户已明确“继续，不需要我再次确认”或同一轮更新范围未变化时，后续批次无需反复确认；遇到扩省、删改既有结论、使用非官方来源等重大变化再停下来说明。
9. 运行校验器。校验失败时修复数据，不能绕过。
10. 每条知识新增或更新都写入 `updateEvents`，再输出本次“新增、更新、替代/废止、待核验、无变化”摘要，并包含所有来源链接。

## 网页预览

常规维护以网页为主要查看入口。除非用户后续重新明确要求单独导出表格，否则不再自动生成、保存或发布 Excel，不在网页提供 Excel 下载按钮，也不向 `docs/downloads/` 写入新的 `.xlsx` 文件。历史已生成的 Excel 文件可保留，不作为后续维护产物。

当用户要求“网页展示”“网页预览”“打开网页看知识库”或类似表达时，生成本地 HTML 网页预览：

~~~bash
node scripts/validate_knowledge_base.mjs --input <知识库路径>/electricity-market.json
node scripts/export_web_preview.mjs --input <知识库路径>/electricity-market.json --output <工作区>/outputs/电力市场知识库-网页预览-YYYY-MM-DD.html
~~~

网页预览必须按固定页签展示：基础概念、全部、国家政策、北京、天津、河北、山西、内蒙古、山东、辽宁、吉林、黑龙江、上海、江苏、浙江、安徽、福建、河南、湖北、湖南、江西、重庆、四川、西藏、陕西、甘肃、青海、宁夏、新疆、广东、广西、海南、贵州、云南、更新记录；省级页签覆盖中国大陆 31 个省级行政区，不含港澳台，并按国网经营区在前、南网经营区在后排序。为优化阅读体验，网页使用紧凑工具栏：搜索框放在页签选择同一行前面，删除“当前页：基础概念”等单独描述区域，尽可能为表格留出高度空间；搜索框和页签固定不随页面滚动，只有表格区域滚动，表头也应保持可见。“全部页签”用于集中检索和展示全国及大陆 31 个省级行政区政策，按政策文件维度汇总国家政策和各省政策，不包含基础概念和更新记录；其字段应在国家政策页字段基础上增加“适用范围”，并按发布日期倒序排序。所有页签表格中的“详细解读”改为“知识摘要”，展示 200 字以内摘要；点击或鼠标悬浮知识摘要时，必须用页面内自定义浮层展示完整知识摘要，移开鼠标、点击空白处或按 Esc 时关闭浮层，不得只依赖浏览器原生 `title` 提示；知识摘要内容若包含 Markdown 样式，必须安全渲染加粗、高亮、行内代码、换行和简单列表，而不是展示原始符号。旧版“政策/规则总结”口径不再作为网页表格主列，统一沉淀为“知识摘要”和完整“详细解读”。每行提供“深度解读”按钮，点击后弹窗只展示完整详细解读，不重复展示其他字段；同时提供“知识浏览”按钮，点击后展示政策正文及附件提取出的 Markdown 渲染内容。政策类页签必须按发布日期倒序排序，最新文件放前面。省份页按政策文件维度展示，字段参考国家政策页，包括“文件标题、知识摘要、发文编号、发布单位、发布日期、链接、查看文件、附件归档、状态、最后核验日期”，网页表格额外提供操作列。详细解读应按文件内容灵活结构化，不使用固定模板；根据政策复杂度设置适合的小标题、条目、重点加粗或高亮，必要时可补充表格或图片等辅助说明。官方来源“链接”必须可点击并支持跳转；“链接”后必须增加“查看文件”字段，点击后在新标签页打开本地归档 PDF；同时展示“附件归档”，点击后在新标签页打开本地附件。网页应提供关键词搜索，便于按概念、政策标题、交易品种、文号、适用范围或主体查找。

网页展示还必须保持整体美学设计：头部使用通过 ChatGPT 图片生成能力制作的电力元素背景图，包含输电塔、电网线、数字化节点等电力市场意象；不得退回纯色背景。头部不展示英文标题，头部不展示副标题说明文案，仅保留中文标题、图标和元信息；中文标题和元信息排版紧凑清晰，避免在 70px 高度内出现拥挤、错位或主次不清。页面整体采用专业、清爽、可信赖的知识库门户风格，头部固定、紧凑工具栏、搜索框和页签固定、表格区域滚动、知识摘要点击/悬浮浮层、Markdown 样式安全渲染、长文本折叠、深度解读弹窗、知识浏览弹窗、链接按钮、搜索框、页签和表格层次需要保持一致的阅读体验。

## 公开发布到 GitHub Pages

用户已确认网页可公开访问，目标仓库为 `wuchaodzxx/electricity-market-knowledge-base`，公开地址为 `https://wuchaodzxx.github.io/electricity-market-knowledge-base/`。发布采用 GitHub Pages 的 `main` 分支 `/docs` 目录：固定首页是 `docs/index.html`，带日期的网页归档仍保留在 `outputs/`。

当用户要求“发布到 GitHub”“更新并发布”“自动发布网页”或类似表达时：

1. 先完成知识库更新、来源文件归档、政策及附件 Markdown 提取、更新记录写入、校验和网页预览导出；不再生成 Excel 下载文件。
2. 运行；常规发布沿用增量 Markdown 提取，默认跳过未变化文件。如确需发布时重新提取全部政策正文和附件，可追加 `--force-extract`：

~~~bash
node scripts/publish_github_pages.mjs --input <知识库路径>/electricity-market.json --outputs-dir <工作区>/outputs --site <工作区>/docs/index.html --date YYYY-MM-DD
~~~

3. 确认 `docs/index.html` 已生成，链接可点击、知识摘要悬浮提示、详情弹窗和知识浏览可用；公开页不提供 Excel 下载入口。
4. 若 GitHub Pages 尚未启用，提示用户在仓库 Settings → Pages 中选择 `Deploy from a branch`，分支为 `main`，目录为 `/docs`；启用后每次推送都会自动公开刷新。
5. 提交并推送到 `origin main`。若本机 GitHub 登录失效或没有推送权限，说明阻塞点并给出重新登录或手动推送步骤；不要假装已发布。
